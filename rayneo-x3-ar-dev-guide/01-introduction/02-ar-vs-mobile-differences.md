# AR 开发与手机开发的核心差异

本文对比手机 App 开发经验，说明在 RayNeo X3 上开发时需要彻底转变的思维模式。**不理解这些差异，你写出的应用在眼镜上几乎无法使用。**

---

## 1. 屏幕模型差异

### 手机开发的假设
- 一块矩形屏幕
- 内容填充整个屏幕区域
- 背景可以是任何颜色

### AR 眼镜的现实

**双屏合目**：逻辑屏幕是两块物理屏幕的并排。你必须把同一份 UI 放在左右两个区域，让两眼看到相同内容。

**透明背景**：纯黑色（`#000000`）在眼镜上显示为**完全透明**。用户通过黑色区域可以直接看到现实世界。这意味着：
- **使用黑色背景 = 透明 = AR 叠加效果**
- **使用白色或其他颜色背景 = 遮挡现实视野**（通常不是你想要的）

```xml
<!-- 正确：AR 叠加风格，黑色区域透明 -->
<ConstraintLayout
    android:background="#000000"  ...>
    <TextView android:textColor="#FFFFFF" ... />
</ConstraintLayout>

<!-- 错误：白色背景会遮挡用户的真实视野 -->
<ConstraintLayout
    android:background="#FFFFFF"  ...>  ← 避免这样做
</ConstraintLayout>
```

**安全边距**：双屏边缘区域在光学合并后可能不舒适或有畸变。SDK 定义了安全边距：
- 水平：`rayneo_safety_padding_horizontal = 30sp`
- 垂直：`rayneo_safety_padding_vertical = 20dp`

核心 UI 内容应在安全边距内。

## 2. 交互模型差异

### 手机开发的假设
- 用户用手指点击屏幕上任意位置
- 触摸事件有精确的 (x, y) 坐标
- Android 系统自动处理焦点（EditText 等）

### AR 眼镜的现实

**没有触摸屏**（眼镜镜片不是触摸屏）。X3 提供了镜腿触控板，**只支持一维输入**：前滑/后滑/上滑/下滑/单击/双击/长按。没有坐标的概念。

这意味着：
- **你不能用 `setOnClickListener` 期待用户"点击"某个 View**
- **所有交互必须通过镜腿手势 + 焦点系统来实现**
- **焦点管理完全由你的应用负责**

#### 交互模型对比

```
手机模式（不适用于眼镜）:
用户 → 点击屏幕 (x=320, y=240) → 系统命中 View → onClick 触发

眼镜模式（正确方式）:
用户 → 滑动镜腿 → SlideForward/Backward 手势 → 焦点切换到下一个 View
用户 → 单击镜腿 → Click 手势 → 当前焦点 View 的 eventHandler 触发
用户 → 双击镜腿 → DoubleClick 手势 → 应用约定：返回/退出
```

#### 约定俗成的手势语义

| 手势 | 推荐语义 |
|------|---------|
| 单击 | 确认 / 选中 |
| 双击 | 返回上一级 / 退出页面 |
| 长按 | 次要操作 |
| 前滑 | 下一个焦点 / 列表向前 |
| 后滑 | 上一个焦点 / 列表向后 |
| 上滑/下滑 | 垂直滚动（X3 独有）|

## 3. UI 设计差异

### 手机开发的假设
- 密集信息展示是正常的
- 多层弹窗、复杂导航树
- 字号 12-16sp 很常见

### AR 眼镜的现实

**视野叠加限制**：眼镜的显示区域是用户视野的一部分，不是全部。UI 必须简洁，不能遮挡太多现实视野。

**字体要更大**：推荐最小字号 24sp 以上，36sp 更佳。小字在眼镜上极难辨认。

**对比度要高**：明亮的文字/图标在黑色背景上，或在半透明深色遮罩上。

**层级要浅**：眼镜操作比手机慢，导航层级越深用户越困惑。建议最多 2-3 层。

**颜色使用 SDK 主题色**：SDK 预定义了 7 个主题色 `color_rayneo_theme_0` 到 `color_rayneo_theme_6`，这些颜色经过光学优化，适合 AR 场景。

## 4. 生命周期与性能差异

### 手机开发的假设
- 应用可以在后台长时间运行
- 屏幕会根据系统自动休眠

### AR 眼镜的现实

**电池限制更敏感**：IMU 传感器、Camera 都是高耗电组件，必须在 `onPause` 时及时释放/注销。

**防休眠需要显式处理**：开发调试时，眼镜可能自动进入休眠影响体验，需要用 ADB 命令或 `WakeLock` 临时禁用（详见调试章节）。

**双屏渲染性能**：`updateView {}` 会让同一套逻辑在左右两份 ViewBinding 上各执行一次，避免在其中做重计算，保持 UI block 轻量。

## 5. 开发调试差异

### 手机开发
- Android Studio 预览器可以直接看效果
- 模拟器可运行大多数应用

### AR 眼镜的现实

**无法用手机模拟器测试**：双屏逻辑、镜腿手势在模拟器上无法正常工作。**必须连接真机调试**。

**双目效果需要实际佩戴**：在屏幕上看到两个并排的矩形 UI 是正常的——这是未佩戴眼镜时的显示，佩戴后双目融合才能看到单一的合目画面。

**scrcpy 投屏只能看单目**：使用特殊参数可以只显示左眼区域（详见调试章节）。

**ADB 是主要调试工具**：日志、截屏、命令都依赖 ADB。`FLogger` 是 SDK 提供的日志工具，推荐使用。

---

## 6. 差异总结对照表

| 维度 | 手机 App | RayNeo X3 AR App |
|------|---------|-----------------|
| 屏幕 | 1 块，任意背景 | 2 块物理屏（1 块逻辑屏），黑色=透明 |
| 触摸 | 任意坐标点击 | 仅镜腿触控板，无坐标 |
| 焦点 | 系统自动管理 | App 完全负责 |
| 导航 | 任意深度层级 | 建议 ≤ 3 层 |
| 字号 | 12sp+ | 24sp+（建议 36sp）|
| 背景色 | 任意 | 黑色（透明叠加现实）|
| 调试 | 模拟器/真机 | 仅真机 |
| 传感器释放 | 可不严格 | onPause 必须释放 |
